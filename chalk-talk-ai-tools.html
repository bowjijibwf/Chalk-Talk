Method

AI 产出质量的三个决定因素

先记住一个通用规则：有效产出 = 模型能力 × 任务上下文 × 推理预算。这是乘法，不是加法。任何一项太弱，整体效果都会明显下降。

有效产出 = 模型能力 × 任务上下文 × 推理预算

从“提示词工程”到 Context Engineering、Harness Engineering，本质都在解决两件事：给足正确上下文，并为复杂任务留出足够推理时间。

1. 模型能力

模型越强，越能理解复杂问题、完成跨步骤分析，也越不需要依赖技巧型提示词去“硬拽”答案。

2. 任务上下文

AI 是基于你给的上下文做推理的。CASE note、附件、官方文档、知识库和 BUG 线索是否准确，直接决定结论质量。

3. 推理预算

TAC 的 CASE 往往不是一句话能答完的问题。复杂排障需要多步检索、交叉验证和工具调用，要给 AI 足够时间。

为什么是乘法

如果模型不够强，上下文再好也推不深；如果上下文不对，再强的模型也会跑偏；如果不给时间，复杂任务只会停留在浅层回答。

现场提醒

后面每个工具都只是把这三件事组合得更好，不会绕开这三个前提。

Tool 1

TAC Copilot：最贴近 CASE 处理流的助手

相信大家多多少少都用过。它最擅长的不是“替你判断”，而是把 CASE note 快速整理成客户更新、Resolution Summary 或 RCA 草稿，把可对外表达的话迅速组织出来。

核心价值

把现有 CASE 信息快速整理成客户可读、结构清晰的正式输出，减少重复写作与摘要整理成本。

典型输出

客户更新邮件、案例摘要、Resolution Summary 或 RCA 草稿，以及把当前记录整理成初步 BUG 线索。（最终内容仍需工程师复核）

使用注意事项

它主要基于 CASE note 做推理。internal note 默认不会自动纳入，需要提前设置或手动打开。

联动上下文

除 CASE note 外，还可联动 RAG、Bugbuster、Topic Search 等数据源，为缺陷线索和相似主题补充上下文。

当前边界

暂时不能直接读取附件；更适合单 CASE 内的整理和重写，不适合多附件、多设备的深度关联分析。

适合优先采用的任务

  • 需要快速形成客户可读更新
  • 需要将分散记录整理为规范摘要
  • 需要对现有案例信息进行结构化重写

边界提醒

  • 默认读 CASE note，不等于自动读 internal note
  • 暂时不能看附件
  • 不是拿来替代工程判断的

Case 1

案例：N9K 更换 PSU 后序列号未刷新

这个版本还比较新，没有找到现成的已知 BUG。TAC Copilot 的价值，在于先把工程上能成立的口径组织出来：序列号可能被缓存；对客户说明时，要把问题性质、workaround 和可选路径讲清楚。

结论（给 TAC / 客户的标准说法）
  • ✅ 这是一个“显示类（cosmetic）问题”，不影响设备运行、供电或冗余能力
  • ✅ 不影响 RMA、功能、稳定性
  • ⚠️ 在不重启设备的情况下，序列号通常无法刷新
原因说明（内部原理，便于解释）
  • 在 Nexus 9000 平台上：PSU 的序列号信息会被缓存。
  • 在 PSU 热插拔场景下：电源的运行状态会实时更新，但 Inventory / SN 信息有时不会重新同步。
  • 该信息通常只在以下情况下刷新：✅ 设备 reload / ✅ 特定 NX-OS 版本修复的 bug。
  • 目前设备运行在 NX-OS 10.3.2，该版本中已观察到类似 PSU SN 不刷新的问题。
使用的 AI 技能与工具演示

调用的工具: Bugbuster Search Defect, Copilot Topic Search

演示链接：点击查看在线演示

案例结论：TAC Copilot 的价值不是替代判断，而是先把可对客表达的话整理好，留给工程师去校验核心技术结论。

观众应记住的重点

  • AI 可以先帮你把“是否有可引用 bug”这件事查清楚。
  • 没有 bug ID 时，仍然可以形成正式、稳健的工程解释。
  • 最终建议应明确问题性质、workaround 和客户可选路径。

Tool 2

Docs-AI：官方文档的权威检索入口

Docs-AI 由文档团队开发，覆盖 CCO 官方文档并实时更新。它的核心价值不是“像搜索一样给链接”，而是给出结论的同时附上引用出处。

产品定位

面向 Cisco 官方文档的问答与检索入口，适合把“结论 + 证据”一起给出来。

核心价值

给出结论、原因和引用出处，让回答更可追溯，也更适合用于技术论证或配置 review。

适用场景

功能支持确认、配置路径核对、版本行为查证、最佳实践引用，以及需要给客户或 BU 附官方依据的场景。

使用注意事项

它只覆盖官方文档，不包含 TZ、Wiki 等内部文档；提问时最好带上产品、版本和功能范围。

官网 docs-ai.cloudapps.cisco.com

公开页面可见产品定位、MCP / REST / Streaming 能力。

Demo / App Try Demo

需要登录。适合现场演示基于官方文档的问答。

适合优先采用的任务

  • 需要基于官方文档给出明确依据
  • 需要快速核对支持矩阵、限制条件与配置路径
  • 需要为其他 Agent 提供权威文档来源

Case 2

案例：BFD Echo 模式是否必须配置 no ip redirects

这个问题很适合现场演示，因为 Docs-AI 会同时给出答案、原因和引用段落，方便把经验判断变成可追溯结论。

现场希望看到的输出
  • 结论：在使用 BFD Echo 模式时，通常需要配置 no ip redirects。
  • 原因：为什么要这样配置，以及不这样做会带来什么影响。
  • 引用：直接给出对应官方文档段落，便于后续复核或引用。
AI 价值
  • 不仅回答“是或不是”，还给出原因和文档出处。
  • 特别适合技术论证、配置审查和后续 case 结论引用。

Tool 3

File Copilot：面向多附件场景的深度分析能力

这是面向多附件场景的深度分析工具，核心就是解决 TAC Copilot 不能读取附件的问题。它能直接读 show tech、日志、pcap、doc、txt，并做跨文件相关性分析。

产品定位

多附件深度分析工具，直接面向 show tech、日志、抓包和文档的联合分析。

关键能力

把分散证据压缩成“初判结论 + 证据 + 下一步动作”，并支持跨文件、跨格式、跨设备建立相关性。

典型输入

show tech、日志、doc、txt、pcap 抓包文件，以及需要联合分析的其他结构化或非结构化附件。

适用场景

多设备协议故障、跨系统事件关联、复杂问题的时间线还原，以及需要输出根因与 next step 的附件分析任务。

使用注意事项

它无法持续对话，所以一次性把已知信息写完整：时间窗口、症状描述、设备范围、拓扑关系、关键附件和你的初步分析。

经验总结

输入越完整，分析结果越接近真实排障过程；重点不是“让它读附件”，而是让它正确理解问题边界和证据范围。

Case 3

案例：N9K 与 ACI 对接的 BFD Echo 抖动分析

File Copilot 把多台设备在同一时间窗口的日志串起来，发现多个节点都记录到 “Echo Function Failed”，初判更像 Echo 回显被对端或共享路径丢弃，而不是单链路控制面超时。

初判结论
  • 抖动的直接原因是 BFD Echo 失败：两台设备都记录了 “Reason: Echo Function Failed”，且时间点一致，说明是 Echo 数据平面回显路径中断，而不是控制平面超时。证据见 logs/5938272d_...
  • OBBPRNX2018 在同一时刻多条 BFD 会话同时 Echo 失败，更像对端（ACI）或共享路径层面的 Echo 不支持/被丢弃，而非单一链路故障。证据见 logs/c9adb398_...
  • BFD 会话曾正常 Up（说明链路和控制面可建立），随后因 Echo 失败 Down，符合“Echo 功能不被对端处理或被策略/硬件丢弃”的模式。
  • 现有配置快照中，受影响的接口在 “show running-config ip” 段落未看到 no ip redirects，如果 BFD echo 启用但缺少 no ip redirects，可能导致 echo 失败。
建议动作
  • 先在对接 ACI 的子接口上禁用 BFD Echo，观察是否消除抖动。
  • 若保留 Echo，需确认接口显式配置 no ip redirects，并适当放宽 BFD 定时器。
  • 并行检查 ACI 侧 BFD / contract、N9K CoPP 以及 Echo / Control 包是否被策略性丢弃。

现场演示

将直接打开工具进行演示，展示AI在提取证据与建议动作上的优势。

点击查看 File Copilot 联合分析过程

案例结论：File Copilot 的价值不只是“读日志”，而是把跨文件证据收束成可执行的假设排序与排障计划。

  • 再对比 AI 的初判和建议动作

Tool 4

SMX：面向领域知识的专家型助手

SMX 是研发团队做的领域专家工具。它的价值在于代码级分析：当问题已经大致收敛到某个组件时，它能把 EDCS 文档、源码级说明和验证命令串起来。

产品定位

把团队 EDCS 文档、源码级知识和 domain tools 做成可检索的专家助手。

典型场景

问题已定位到某个模块后，继续查相关 C 代码、组件交互、验证命令和版本差异。

核心价值

适合从“问题在哪个组件”继续往下走到“这段代码/这条路径在做什么”，尤其适合多组件协作场景。

使用注意事项

平台和版本差异仍需人工判断；SMX 提供的是更深的工程解释，不是万能问答。

官网 smx.cisco.com

页面能直接看到定位、特性和 GitHub Copilot 集成信息。

使用入口 SMX Chat Utility

需要登录。适合现场打开演示已有 domain。

适合优先采用的任务

  • 需要特定团队或模块的长期知识
  • 需要代码级或流程级解释
  • 需要将私有经验沉淀为可复用资产

Case 4

案例：vPC 环境下 ARP 是否同步、是否实时

这个问题看起来简单，真正难点是不同平台、不同版本的行为并不完全一样。SMX 的价值，是把源码片段、内部说明和验证命令拼成一个工程解释。

为什么这个问题不好答
  • 它不是一个简单的 yes / no 问题。
  • 不同平台、不同版本的行为可能有差异。
  • 公开文档通常不会把内部组件关系讲得很透。
SMX 给出的工程解释
  • NX-OS 在 vPC 环境下有 ARP 同步机制。
  • 同步是实时进行的，而不是只在重启时触发。
  • 可以结合 arp_mcec.c 相关说明和验证命令形成工程口径。

现场演示

链接：点击查看 SMX 问答对话

观众应记住的重点

  • SMX 适合回答“公开文档不完整，但团队内部长期知道”的问题。
  • 它能把源码级文档、架构知识和验证命令组合成完整解释。
  • 这类回答特别适合模块 owner、平台专家和深度排障场景。

Tool 5

GitHub Copilot + MCP：可定制的工作流引擎

如果前面几个工具更像“单点能力”，这里更像一个可以自己编排的工作台：按需接入数据源，自由切换模型，注入提示词和 Skills，把高频复杂任务沉淀成可复用工作流。

1. 运行环境与入口

VS Code 里可以直接演示运行模式、模型选择、Session、Skills 和 MCP，把“对话”变成“可执行工作流”。

2. 数据源与模型

按需接入文档、Bug、Topic Search、历史经验等数据源；再根据任务复杂度切换模型，兼顾速度、成本和推理能力。

3. Skills 的价值

适合沉淀高频复杂任务，例如给 CASE 分析固定命令、检查顺序和输出格式。

4. 使用提醒

要考虑数据治理、接入成本、模型选型和 Skills 维护；不是所有问题都需要 Skills。

做 Skills 的经验

  • 不要给 AI 它本来就会的东西
  • 只注入需要定制的命令、流程和 Cisco 风格
  • Skills 太大、太杂，效果反而会下降

现场演示会看什么

  • 运行模式与模型选择
  • Session、Skills 与 MCP
  • 两个复杂案例如何被编排

Case 5

案例：复杂拓扑下的丢包路径分析

这是我“知道怎么做，但纯手工会花大半天”的那类问题。重点不是单点命令，而是要在多设备、多抓包、多段路径之间把证据串起来。

为什么耗时

要逐点核对 request_id、MAC/IP 映射、去程回程路径，还要串联十台设备、服务器侧和负载均衡两侧的信息。

人工也能做，但会耗费大量时间，而且非常容易在碎片信息之间漏掉关键关联。

Agent 帮你省下什么
  • 在文本、抓包和多设备信息之间来回切换
  • 快速判断丢包发生在去程还是回程
  • 把零散证据组织成可交付的 RCA
案例结论：AI 真正替代的是那些依赖堆人力、耗时长、极易遗漏碎片的“大海捞针”和“数据串联”工作。

现场互动

  • 先请一位同学讲讲他的排查思路
  • 再展示 Agent 如何补齐证据链
  • 最后对比人工和 Agent 的时间成本

Case 6

案例：50000 个包里找 1 个丢失的

这是我拿到问题也会先挠头的场景：50001 个数据包里只丢 1 个，先要找出是哪一个、精确到什么时间点，再去对齐那一刻的控制面日志。

问题难点
  • 50001 个数据包（49999 个 ICMP）里只丢了一个。
  • 要先找出具体丢的是哪个 sequence、精确到什么时间点。
  • 找到时间点之后，还要回到日志里解释它为什么会丢。
Agent 后续怎么继续
  1. 找到精确丢失包和时间点。
  2. 对齐对应时刻的控制面和组表日志。
  3. 继续分析动态/静态组表项交互对转发的影响。

分析难点突破

人工统计和排查会耗费大量时间。通过 Custom Agent 输出代码/执行分析，不但能极快定位断包点，还能连贯完成后续对动态与静态组表项交互影响（M2RIB/PSS OIF）的根因推理。

Closing & Summary

回顾与总结：按任务特点灵活选型

最后快速回顾三点：先看通用框架，再看五类工具的适用范围，最后别忘了 TAC 工程师在 AI 流程中的责任。

今天要带走的 3 个点

1. 有效产出 = 模型能力 × 任务上下文 × 推理预算

2. 五类工具分别对应重写摘要、官方文档、多附件分析、领域专家和 Agent 编排

3. AI 负责提速，TAC 工程师负责审查方向、结论和证据

按任务选刀

日常沟通重写与基础摘要优先用 TAC Copilot；确认文档依据优先查 Docs-AI；多附件压缩与交叉对照直接上 File Copilot；内部长期代码/架构解惑向 SMX 提问；跨系统长链路分析再用 GitHub Copilot + MCP 组工作流。

一句话总结

不是某一个工具最强，而是遇到什么问题用什么工具。

工程闭环

  • 看方向是不是对的
  • 看结论是不是站得住
  • 看证据是不是足够支撑
