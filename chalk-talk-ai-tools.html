Method

AI 产出质量的三个决定因素

无论使用哪一类工具，最终效果都取决于三个前提：模型能力是否足够、任务上下文是否准确、系统是否获得了完成推理所需的时间与工具。

有效产出 = 模型能力 × 任务上下文 × 推理预算

在 TAC 场景中，失败往往不是由单一因素造成，而是源于三者中的至少一项配置不足。后续各工具不再重复展开这一原则，而是聚焦其最关键的使用注意事项。

1. 模型能力

决定系统是否能够理解复杂问题、完成跨步骤分析，并产出可直接使用的结果。

2. 任务上下文

决定系统是否基于正确材料开展推理。CASE note、官方文档、附件日志、知识库与 BUG 数据均属于上下文来源。

3. 推理预算

决定系统能否完成多步检索、跨文件关联与工具调用。复杂排障任务通常需要更长的推理链和更完整的执行过程。

后续呈现方式

后续每一类工具主要回答两个问题：

  • 适合解决哪一类 TAC 任务
  • 实际使用时需要注意什么

核心结论

工具能力的上限，往往由输入材料质量和任务定义质量共同决定。

Capability Map

五类工具的能力梯度与选型关系

这五类工具并非替代关系，而是从贴近案例处理到高度定制化能力的递进关系：先解决内容生成与文档核对，再进入多附件分析、领域专家知识与 Agent 编排。

1
近场提效

TAC Copilot

最贴近当前 CASE 流程，适合将 note 转化为客户可读的正式输出。

2
权威检索

Docs-AI

最适合回答“官方文档如何定义该功能、行为或限制”。

3
深度分析

File Copilot

最适合跨附件、多格式和多设备的相关性分析与故障归因。

4
领域专家

SMX

最适合依赖团队私有知识、内部流程或代码级理解的场景。

5
定制 Agent

GitHub Copilot + MCP

将模型、工具与数据源编排成可复用的高级工作流与执行能力。

选型原则

优先选择与任务距离最近、且已具备足够上下文的工具。没有必要用最复杂的 Agent 处理最简单的输出任务。

Tool 1

TAC Copilot：最贴近 CASE 处理流的助手

TAC Copilot 面向 TAC 工程师提供基于 CASE note 的内容生成能力，可辅助形成客户响应、GCI、RCA 和 Resolution Summary 等输出。

核心价值

将现有 CASE 信息快速整理为客户可读、结构清晰的正式输出，减少重复写作与摘要整理成本。

典型场景与典型输出

快速生成客户更新邮件、案例摘要、case audit、Resolution Summary 或 GCI / RCA 草稿，并将现有记录整理为初步 BUG 线索。（最终内容仍需工程师复核与调整）

使用注意事项

其主要推理基础是 CASE note。默认不会纳入 internal note，因此需要提前将可对外表达的关键事实、结论和下一步计划整理到有效 note 中。

联动数据能力

除 CASE note 外，可联动 RAG 数据、Bugbuster、Topic Search 与 BORG 签名，用于补充缺陷线索、相似主题和平台特征提示。

边界

适合单案例内的内容生成与缺陷线索整理，不适合承担多设备、多附件的复杂根因关联分析。

适合优先采用的任务

  • 需要快速形成客户可读更新
  • 需要将分散记录整理为规范摘要
  • 需要对现有案例信息进行结构化重写

案例：N9K 更换 PSU 后序列号未刷新

TAC Copilot 联动 RAG、Bugbuster、Topic Search 与 BORG 签名后，快速完成缺陷排查与 SR 收口建议。这个案例的关键不在于“找到一个 bug ID”，而在于形成可直接对客户解释的工程结论。

结论（给 TAC / 客户的标准说法）
  • ✅ 这是一个“显示类（cosmetic）问题”，不影响设备运行、供电或冗余能力
  • ✅ 不影响 RMA、功能、稳定性
  • ⚠️ 在不重启设备的情况下，序列号通常无法刷新
原因说明（内部原理，便于解释）
  • 在 Nexus 9000 平台上：PSU 的序列号信息会被缓存。
  • 在 PSU 热插拔场景下：电源的运行状态会实时更新，但 Inventory / SN 信息有时不会重新同步。
  • 该信息通常只在以下情况下刷新：✅ 设备 reload / ✅ 特定 NX-OS 版本修复的 bug。
  • 目前设备运行在 NX-OS 10.3.2，该版本中已观察到类似 PSU SN 不刷新的问题。

Case 2

案例：BFD Echo 模式是否必须配置 no ip redirects

Docs-AI 的典型价值在于：不仅给出结论，还能同时附上依据来源。对于“BFD Echo 是否必须关闭 redirect”这一类问题，官方文档型答案比经验型回答更容易形成可复核结论。

答案要点
  • 在启用 BFD Echo 模式前，需要配置 no ip redirects，以避免高 CPU 利用率问题。
  • BFD Echo 依赖转发平面回显机制，不适合与 uRPF 同时启用，否则会造成会话 flap。
  • 这一结论在 Cisco IOS 与 Catalyst 相关 BFD 文档中保持一致。
AI 价值
  • 不仅回答“是或不是”，还给出原因、约束条件和文档出处。
  • 适合直接用于技术论证、配置 review 和后续 case 结论引用。

Tool 3

File Copilot：面向多附件场景的深度分析能力

File Copilot 直接面向 case 附件开展开放式分析，可跨文件、跨格式、跨设备建立相关性，是从“手工摘要输入”走向“原始材料直接分析”的重要升级。

产品定位

当前处于 pilot 阶段，可使用 OpenAI Codex agent 与 GPT-5 模型对 case 附件开展通用分析，并生成问题总结、根因判断与建议动作。

关键能力

开放式分析、多文件关联、跨格式联合分析，以及结合 Borg IC 结果的综合判断。

典型输入

show tech、日志、doc、txt、pcap 抓包文件，以及需要联合分析的其他结构化或非结构化附件。

适用场景

多设备协议故障、跨系统事件关联、复杂问题的时间线还原，以及需要输出根因与 next step 的附件分析任务。

使用注意事项

一次性提供足够的故障材料至关重要，包括时间窗口、症状描述、设备范围、拓扑关系和关键附件。输入越完整，分析结果越接近真实排障过程。

分析方式

可配合领域 skill 使用，例如 nexus_dcrs_tech_support_bundle_analyzer，将 show tech、BFD 日志和平台经验结合为结构化分析过程。

Case 3

案例：N9K 与 ACI 对接的 BFD Echo 抖动分析

通过 nexus_dcrs_tech_support_bundle_analyzer 对 N9K show tech 与 BFD 日志进行联合分析，File Copilot 将多份日志压缩成“初判结论 + 证据 + 下一步动作”的工程输出。

初判结论
  • 抖动的直接原因是 BFD Echo 失败：两台设备都记录了 “Reason: Echo Function Failed”，且时间点一致，说明是 Echo 数据平面回显路径中断，而不是控制平面超时。证据见 logs/5938272d_...
  • OBBPRNX2018 在同一时刻多条 BFD 会话同时 Echo 失败，更像对端（ACI）或共享路径层面的 Echo 不支持/被丢弃，而非单一链路故障。证据见 logs/c9adb398_...
  • BFD 会话曾正常 Up（说明链路和控制面可建立），随后因 Echo 失败 Down，符合“Echo 功能不被对端处理或被策略/硬件丢弃”的模式。
  • 现有配置快照中，受影响的接口在 “show running-config ip” 段落未看到 no ip redirects，如果 BFD echo 启用但缺少 no ip redirects，可能导致 echo 失败。
建议动作
  • 先在对接 ACI 的子接口上禁用 BFD Echo，观察是否消除抖动。
  • 若保留 Echo，需确认接口显式配置 no ip redirects，并适当放宽 BFD 定时器。
  • 并行检查 ACI 侧 BFD / contract、N9K CoPP 以及 Echo / Control 包是否被策略性丢弃。

Tool 4

SMX：面向领域知识的专家型助手

SMX 是面向特定技术域的 AI assistant framework，可将团队文档、知识、指令与工具整合为持续演进的领域专家，并与 GitHub Copilot 等入口配合使用。

产品定位

面向特定技术域的专家助手框架，支持文档摄取、自学习、custom tools 和 inter-SMX collaboration。

典型场景

代码级专家帮助、领域知识问答、runbook 咨询、新人 onboarding，以及将团队私有经验沉淀为持续可访问的专家能力。

Case 4

案例：vPC 环境下 ARP 是否同步、是否实时

SMX 的价值体现在“源码级和团队级知识的解释能力”。这个案例不是单纯回答 yes / no，而是把 NX-OS 内部组件、同步模式和验证命令一起组织成工程解释。

我的问题 1：vPC 环境下 ARP 有同步机制吗？

AI的回答：是的，NX-OS 在 vPC 环境下有 ARP 同步机制。 根据检索到的源码级文档，ARP 进程中存在专门的 MCEC/vPC 路径（arp_mcec.c），并且 ARP 进程链接了 libmcecm_stub 库，这是 MCEC（Multi-Chassis EtherChannel，即 vPC 的内部框架名）的核心集成模块。

我的问题 2：这个同步会实时进行还是只在重启时候？

AI的回答：## ARP 同步是实时进行的

根据检索到的 NX-OS 源码架构文档，vPC 环境下的 ARP 同步是**实时（real-time）进行的**，而非仅在重启时触发。

Tool 5

GitHub Copilot：走向灵活定制与深度推理的 Agent

不再仅仅是对话工具，通过灵活定制数据源、模型选型、提示词注入与 Skills 挂载，形成可持续自我改进与深度推理的工作流。

1. 数据源 (MCP / Data Sources) 的按需接入

不仅仅局限于本地文件，可灵活接入内部 BUG 数据、历史 SR、团队私有 Wiki 与 Docs-AI。打通内部外系统壁垒，让 Agent 获得最新、最权威的业务上下文。

2. 模型选型 (Model Selection) 的自由切换

针对逻辑复杂的疑难问题可选 GPT-5.4 或 Opus 4.6 等高推理模型，针对基础长文本处理可快速切回 Haiku 4.6 或 Gemini，兼顾成本、速度与准确度。

3. 提示词与 Skills (Prompt & Skills) 的工程沉淀

通过自定义 .instructions 文件或挂载特定域的 SKILL（如 nxos-universal-triage），强制 Agent 在分析前载入领域专家的检查清单和经验，避免幻觉与泛泛而谈。

4. 持续提醒与高阶推理能力 (Reasoning)

让 AI 懂得“三思而后行”。在长上下文中，AI 能持续记住你设定的推理纪律、分析边界与必须回答的具体疑问，有效整理复杂多设备的碎片日志，实现多步自主推理。

适合优先采用的任务

  • 需要跨多个系统整合证据
  • 需要统一编排工具调用与输出格式
  • 需要把高频复杂任务固化为可复用流程

案例主题：29.2.8.50 周期性 SNMP 访问丢包 RCA

问题背景：涉及 10 台设备、服务器侧与负载均衡的杂乱环境。我们需要确认跨界丢包发生在去程还是回程，在哪一段非对称路径丢弃，并形成报告。

人的困境：逐点核对 request_id、整理 MAC/IP 映射、串联十个文件的信息虽然也能做，但将耗费大量时间并且极易出错。

  • 展示在 VS Code 中调用模型结合 MCP 分析，完成完整 RCA

观众应记住的重点

  • 它不仅能“找信息”，还能“组织证据”。
  • 它不仅能“看一份文件”，还能“串多份文件”。
  • 它特别适合复杂 RCA 和高频分析流程沉淀。

Case 6

案例：组播随机丢包的多步排障与高阶定位

这个案例展示了如何在 Agent 的辅助下，从 50,001 个数据包中找出一包丢失的时间点，进而联动分析特定时刻的控制面日志。

问题背景与范围
  • 概要: 两台新接入的 N93180 交换机上，业务流 `(10.140.50.93, 225.0.2.189)` 出现不定期丢包。客户抓包确认收到流量，但故障时未能从本地下联接收口发出。
  • 范围: 两台 N9K 下 VLAN 850 内的本地接收者；H42 重点接收口为 `Ethernet1/22`。
  • 症状: 两台丢包不同步；`Eth1/22` 增加静态组仍丢包；后客户关闭恒生 NSQ 冷备后恢复。
客户的核心疑问
  1. 为什么静态组加上后仍丢包？
  2. 在 LAB 重现的 50001 个数据包 (49999 个 ICMP) 中，到底丢了哪一个 sequence？如何找到精确到秒的丢失时间？

分析难点突破

人工统计和排查会耗费大量时间。通过 Custom Agent 输出代码/执行分析，不但能极快定位断包点，还能连贯完成后续对动态与静态组表项交互影响（M2RIB/PSS OIF）的根因推理。

Closing & Summary

回顾与总结：按任务特点灵活选型

对于 TAC 团队，五类工具更适合被理解为一个工具箱：先解决高频低门槛任务，再逐步向纵深复杂排障与灵活 Agent 编排演进。

核心选型建议

1. 日常沟通重写与基础摘要：优先用 TAC Copilot

2. 确认指标、命令与支持矩阵：优先查 Docs-AI

3. 原始多附件压缩、交叉对照：直接上 File Copilot

4. 内部长期代码/架构解惑：向 SMX 提问

5. 长文本复杂推理、持续记忆与跨域：用 GitHub Copilot + Skills 构建工作流

关键结论

真正产生稳定价值的不是单一工具本身，而是“遇到什么问题用什么刀”，并辅以后置的人工验证闭环。将工具箱用顺，就是 AI Empowered TAC 的关键。
